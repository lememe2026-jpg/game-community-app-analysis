# 项目修订与学习日志

本文用于记录本项目从生成、审稿、重写到公开仓库精简过程中的主要修订思路和学习点。

## 项目定位

本项目是个人学习、行业理解和求职准备过程中的一次尝试性公开数据分析项目。它不是商业咨询报告，也不是行业研究结论，更不是对小黑盒、游民星空或相关公司真实运营能力的评价。

项目目标是通过公开数据练习：

- 数据清洗与基础分析。
- 游戏社区 App 行业理解。
- 用户评论、关键词、下载预估、版本日志的交叉观察。
- Python 脚本阅读与结果复核。
- Excel 复核与 SQL 后续学习。
- AI / Codex 辅助下的代码理解、文档审校和项目整理。

## 修订过程概览

项目经历了以下几个阶段：

- AI / Codex 辅助生成初版分析结构和说明文档。
- 人工审稿，识别过强结论、模板化运营建议和“AI 味”表达。
- 手动重写正式报告，把口径从求职包装和运营建议清单收束为公开数据观察。
- 检查 PDF 展示效果，确认正式报告页数、图表和表格显示状态。
- 对齐 README 与 docs，统一说明项目是个人学习型尝试项目。
- 精简公开仓库，只保留报告、PPT、汇总表、图表、脚本和必要说明文档。

## 主要学习点

本轮整理的重点不是新增分析，而是学习如何控制公开数据项目的边界：

- 公开评论只能说明样本中出现过相关表达，不能代表全部用户观点。
- 关键词搜索指数不能写成真实流量或真实用户规模。
- 七麦 iPhone 新增下载预估是第三方估算值，不代表官方真实下载。
- 版本日志只能作为公开动作背景，不能证明功能效果。
- TGA 等外部事件窗口只能作为外部观察变量，不能写成下载增长原因。
- 四源数据可以形成交叉观察，但不能组成因果证明链。

## 降低 AI 味和模板化表达

早期文本中容易出现过于正式、像商业咨询报告或运营 SOP 的表达，例如：

- “证明增长”
- “导致下载提升”
- “运营能力强弱”
- “活动承接效率”
- “功能动作带来效果”
- “公司真实运营能力判断”

后续修订将这些表达降级为：

- “提示观察窗口”
- “作为公开动作背景”
- “可以形成交叉观察”
- “需要结合内部数据验证”
- “公开数据不足以完成因果归因”

这次修订让我更明确：公开仓库展示的价值不在于把结论写得很强，而在于把数据来源、解释边界、复核过程和学习过程说清楚。

## 人工修改与 AI 辅助关系

AI / Codex 在项目中主要辅助代码阅读、脚本解释、文档草稿、格式清理、路径检查和口径风险检查。项目选题、报告手动修改、证据边界、关键结论取舍和最终确认由本人完成。

因此，本项目不应被描述为 AI 独立完成，也不应被描述为本人已经熟练掌握全部代码、SQL 或完整数据工程流程。更准确的定位是：在 AI 辅助下完成的一次学习型公开数据分析实践。

## 公开仓库整理原则

公开仓库只保留面向阅读和复核必要的文件：

- `README.md`
- `portfolio/`
- `slides/`
- `docs/`
- `outputs/charts/`
- `outputs/tables/`
- `scripts/`
- `requirements.txt`

公开仓库不保留完整 raw 数据、归档目录、备份目录、旧版报告、旧版 PPT、过程检查报告、清理日志、临时 HTML 或 `.bak` 文件。

这样做的目的不是隐藏分析过程，而是避免把包含评论文本、平台导出文件、旧草稿和本地清理痕迹的内容放入公开仓库，同时让读者更容易理解正式报告、汇总结果和项目边界。

## 后续学习方向

后续可以继续做的事情包括：

- 使用 SQL 复刻评论、关键词、下载和版本的关键汇总表。
- 为主要脚本补充更清楚的阅读笔记。
- 在不公开 raw 数据的前提下，补充字段口径说明。
- 如果有真实业务数据，再学习如何验证公开观察中提出的问题线索。
- 持续练习把 AI 辅助输出转化为经过人工判断和复核的个人项目表达。
